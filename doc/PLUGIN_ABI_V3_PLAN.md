# YiCAD 插件 ABI v3 发布计划

## 1. 文档定位

本文是 ABI v3 从当前草案进入正式发布的执行计划，不是 ABI 契约。公开契约以
`YiCadPluginAbi.h`、`PLUGIN_ABI_EVOLUTION.md` 和 `PLUGIN_SDK.md` 为准。

当前代码基线为 `develop` 分支提交 `7e49d2a`。阶段 0–7 已完成，阶段 8、9 尚未完成：

| 阶段 | 状态 | 结果 |
| --- | --- | --- |
| 0–6 | 已完成 | 导入会话、资源、实体和第一版 SDK 已实现。 |
| 7 | 已完成 | ABI 布局重构、高层 SDK、demo 迁移及编译期布局门禁已实现。 |
| 8 | 待实施 | 建立可重复的 ABI、SDK、宿主运行时兼容性验证并关闭审查问题。 |
| 9 | 待实施 | 冻结 ABI v3，解除草案隔离并发布 SDK。 |

在阶段 9 完成前必须保持以下状态：

- `YICAD_PLUGIN_ABI_MAX_VERSION == YICAD_PLUGIN_ABI_V2`；
- v3 仅在 `YICAD_ENABLE_PLUGIN_ABI_V3_DRAFT` 开启时参与开发构建；
- 安装后的 SDK 不传播草案宏，不得用于发布声称支持 ABI v3 的插件；
- 阶段 8 可以修复 v3 草案布局，但不得修改已发布的 v1/v2 前缀；
- 阶段 9 只做发布收口。若仍需改变布局或行为，必须退回阶段 8 重新验证。

### 1.1 `structSize` 的实际状态

普通 C++ SDK 和 demo 中已经没有手工填写 `structSize` 的要求。SDK 通过默认构造、语义
类型和 builder 生成临时 C ABI POD，插件业务代码只提供资源和实体数据。

底层 C ABI 并未完全删除 `structSize`，阶段 8 不得按“全局移除”处理：

- `YiCadStringView`、点、向量、颜色、基础数组视图和 `YiCadVertex2d` 等固定布局值类型
  不含 `structSize`；
- v3 可扩展输入结构仍以 `structSize` 描述实际可访问前缀；这是旧前缀兼容机制；
- `YiCadHostApi`、`YiCadPluginApi` 的该字段属于已发布 v1 前缀，不能删除；
- `YiCadImportApi` 使用同一函数表裁剪规则；
- `byteStride` 只存在于可扩展的 HATCH 环/边数组视图，由 SDK 自动维护。

因此，发布验收目标是“常规 SDK 用户看不到 ABI 记账字段，底层 ABI 能正确验证它们”，
不是从公开 C ABI 中删除这些字段。

## 2. v3 发布范围

### 2.1 目标

ABI v3 为高保真文件导入提供与格式无关的宿主写入接口。插件解析 DXF、SVG、DGN 等外部
格式，宿主负责创建 YiCAD 文档资源、实体和块。公开接口不得暴露 `Dm*`、Qt、STL 或
`libdxfrw` 类型。

本版本冻结以下能力：

- 文档设置：插入单位、测量制式、全局线型比例和源代码页元数据；
- 资源：图层、简单线型、文字样式和标注样式；
- 公共属性：图层、线型、颜色、线宽、线型比例、可见性和法向量；
- 基础实体：POINT、LINE、RAY、XLINE、ARC、CIRCLE、ELLIPSE、二维 POLYLINE、SPLINE；
- 语义实体：TEXT、MTEXT、BLOCK、INSERT、ATTDEF、ATTRIB、DIMENSION、LEADER、HATCH、IMAGE；
- 原子导入、单步撤销、失败回滚、诊断结果和会话内句柄。

对宿主模型不能保真表达的输入只能采用以下策略之一：

- 明确返回 `YICAD_IMPORT_ERROR_UNSUPPORTED`，且不修改文档；
- 仅在调用方显式允许时降级，并产生可查询诊断；
- 在阶段 9 前补齐模型和映射，改为完整支持。

复杂线型、三维多段线/网格、有理或周期样条、不可表达的 MTEXT 控制码、坐标标注和图像
裁剪边界必须在阶段 8 逐项核对现有实现与上述策略，禁止静默丢字段。

### 2.2 非目标

以下内容不进入 ABI v3：

- 完整高保真导出和任意实体只读枚举；
- 图纸空间、布局、视口和页面设置；
- 动态块、约束、MLEADER、TABLE、MLINE 和垂直产品代理对象；
- 插件自定义实体、跨 DLL C++ 对象继承、3D 面/实体/曲面；
- 后台线程修改文档；
- 面向具体文件格式的解析器或数据类型。

本轮不新增 CTest、`tests/` 目录、专用测试插件、测试夹具、测试数据或测试脚本。发布门禁
由生产头文件中的编译期断言、完整构建与安装、现有 demo、真实插件人工验收和代码审查
组成；不得为便于验证而扩大公开 ABI 或加入仅供验证使用的生产分支。

## 3. 已冻结的候选设计

### 3.1 版本协商和函数表

- 插件导出 `yicad_plugin_get_abi_version()`，返回自身实现的最高版本；
- 宿主使用 `min(插件最高版本, 宿主最高版本)` 作为协商版本；
- v1/v2 的 `YiCadHostApi` 前缀保持不变；v3 只在尾部追加一个 `importApi` 指针；
- `YiCadImportApi` 独立承载全部导入能力，缺失尾字段或空函数指针只表示该能力不可用；
- 插件必须在 `YiCadPluginApi::abiVersion` 中原值确认协商结果，否则加载失败并回滚注册；
- v3 插件面对 v2 宿主时可以按 v2 初始化，也可以从 init 明确拒绝，不允许误用 v3 字段。

### 3.2 导入会话和所有权

```text
beginImport(document)
  -> 创建资源、块和实体
  -> commitImport(session)    成功，形成至多一个撤销项
  -> rollbackImport(session)  失败或取消，不保留部分结果
```

- 同一文档最多有一个活动导入会话，且不得与既有宿主/插件事务嵌套；
- `commitImport` 和 `rollbackImport` 都会消费会话并使全部子句柄失效；
- 空会话提交不得产生无意义撤销项；
- 文档关闭、插件卸载和宿主销毁必须回滚仍活动的会话；
- 容器和资源句柄仅在所属活动会话中有效，插件不得释放或跨回调缓存；
- 插件传入的字符串、数组和嵌套指针只在调用期间借用，宿主在返回前复制所需数据；
- 不跨 DLL 分配/释放内存，不让 C++ 异常穿过 C ABI；
- v3 写接口仅允许在创建宿主 API 的 UI 线程调用。

### 3.3 布局规则

1. 固定布局值类型按值传递，不原地扩展；需要新增信息时定义新类型。
2. 可扩展输入必须独立传入或通过指针借用，只能在尾部追加字段。
3. 可扩展结构不得按值嵌套另一个可扩展结构；空嵌套指针必须具有唯一语义。
4. 宿主按调用方声明的可访问前缀读取字段，接受合法旧前缀并忽略未知尾字段。
5. 缺失的可选尾字段使用公开默认值；当前必需前缀由唯一的 `YICAD_*_MIN_SIZE` 定义。
6. 可扩展数组必须提供字节步长，并在解引用前检查指针、数量、最小前缀、对齐和溢出。
7. 字符串统一为 UTF-8 显式长度视图，不允许嵌入 NUL；所有浮点输入必须有限。
8. 枚举值、布尔值、资源类型、容器类型和会话归属必须在使用前验证。

具体字段顺序、大小、对齐和偏移不在本文重复维护。阶段 8 以前的候选真值由
`YiCadPluginAbi.h` 中的类型、`YICAD_*_MIN_SIZE` 和静态断言共同定义；阶段 9 将其转为
正式冻结契约。

### 3.4 SDK 边界

- 常规入口使用 `ImportSession`、`ImportContainer`、`ImportResource` 和领域数据类型；
- SDK 对象拥有插件侧字符串和容器，仅在调用时生成临时 C ABI 视图；
- SDK 自动生成公共属性默认值、嵌套指针、数组数量和步长；
- 裸 POD 入口只用于其他语言绑定、高级接入和 ABI 验证；
- 高层入口不得改变宿主结果码、诊断和降级语义。

## 5. 阶段 8：发布候选验证与加固

目标：在不新增测试代码的前提下完成 ABI 边界审计、生产代码加固和发布候选验收。本阶段
结束时 v3 仍是草案；任何布局或语义修复都必须重新执行本阶段全部门禁。

### 8.1 交付边界

阶段 8 只允许修改生产头文件、宿主实现、SDK、demo、构建安装规则和公开文档。不得新增
CTest、测试目录、专用验证插件或仅供验收使用的代码路径。验证记录保存在发布说明或评审
记录中，不作为运行时代码进入安装包。

### 8.2 ABI 布局和语言边界

生产头文件中的 `static_assert` 必须覆盖：

- v1/v2 函数表的大小、对齐、字段偏移、精确函数类型和 `__cdecl` 调用约定；
- 草案开启后 `YiCadHostApi::importApi` 只追加在 v2 前缀尾部；
- Win32/Win64 固定布局值类型的大小、对齐和关键偏移；
- 每个可扩展输入的 `YICAD_*_MIN_SIZE` 与必需前缀末字段一致；
- `YiCadImportApi` 的字段顺序、函数类型、大小和对齐；
- 插件入口的 C 链接和调用约定。

公开头文件必须同时满足以下边界：未定义草案宏时可独立包含且不泄漏 v3 类型；按 C11
解析时不依赖 C++、Qt 或 STL；按 C++23 使用时不改变 C ABI。语言边界检查直接对公开头
文件执行编译器语法检查，不创建额外源文件。

如果无法实际构建 Win32，必须在阶段 8 把正式支持范围收缩为 Win64，并同步修改 SDK 文档
和发布说明。未执行的 Win32 断言不能作为支持 Win32 的依据。

### 8.3 版本协商和函数表审查

逐条追踪插件加载生产代码，确认以下矩阵：

| 插件声明 | 宿主能力 | 期望结果 |
| --- | --- | --- |
| v1 | v3 | 协商 v1，只暴露 v1 宿主前缀。 |
| v2 | v3 | 协商 v2，只暴露 v2 宿主前缀。 |
| v3 | v3 | 协商 v3，且 `importApi` 可用。 |
| v3 | v2 | 协商 v2；插件确认则加载，拒绝则回滚注册。 |
| 高于 v3 | v3 | 协商 v3，插件必须原值确认。 |
| 0 或低于最低版本 | 任意 | 在 init 前拒绝。 |

审查还必须覆盖宿主表和导入子表被合法裁剪、尾函数指针为空、插件输出表过短、插件回报
版本不一致，以及 init 失败或抛出异常的路径。每条失败路径必须撤销本次注册，按既有生命
周期清理插件，不遗留活动会话或回调。

### 8.4 输入解码和运行时加固

所有 v3 入口统一复用生产解码与校验函数，不允许各实体自行解释 ABI 前缀。代码审查至少
确认以下规则已落实：

- 只读取 `structSize` 覆盖的字段，忽略未知尾字段，缺失可选尾字段使用公开默认值；
- 必需指针、枚举、布尔值、有限浮点数、数值范围和 UTF-8 字符串在写入模型前完成校验；
- 句柄同时校验类型、所属会话和生命周期；
- 数组在解引用前校验数据指针、数量、步长、对齐以及乘加溢出；
- HATCH、SPLINE、POLYLINE、块引用和资源依赖先完成整体一致性检查，再修改文档；
- 不支持能力返回 `YICAD_IMPORT_ERROR_UNSUPPORTED`；显式降级必须产生诊断；
- C ABI 最外层捕获全部异常，并转换为稳定结果码。

不得为某个实体复制一套局部校验逻辑。发现重复规则时应先收口内部实现，再继续发布审查。

### 8.5 会话、撤销和清理

使用现有 demo 和真实文档完成以下人工验收，不增加专用入口或夹具：

- 成功导入至多产生一个撤销项，撤销后恢复导入前状态；
- 空会话提交不产生撤销项；
- 显式回滚、SDK RAII 回滚和中途失败不留下部分文档状态；
- 同一文档的第二个会话、事务嵌套和重复提交/回滚被拒绝；
- 提交或回滚后，会话、容器和资源句柄立即失效；
- 文档关闭、插件卸载和宿主退出会回滚活动会话；
- 插件与宿主之间不存在跨模块释放或异常传播。

若现有 demo 无法触达某条失败路径，只做生产代码审查并记录限制，不得添加测试专用开关。
任何无法通过代码证明的布局、所有权或生命周期问题都必须阻止发布。

### 8.6 构建、安装与人工验收矩阵

阶段 8 至少完成以下配置：

| 架构 | 配置 | 草案 | 必须产物与检查 |
| --- | --- | --- | --- |
| Win64 | Debug | OFF | 宿主和正式 v1/v2 PluginSDK；安装包不含 v3 草案定义。 |
| Win64 | Release | OFF | 宿主和正式 v1/v2 PluginSDK；现有外部插件可继续构建、加载。 |
| Win64 | Debug | ON | v3 宿主和现有 demo；完成调试配置人工验收。 |
| Win64 | Release | ON | v3 宿主和现有 demo；完成安装目录人工验收。 |
| Win32 | Debug/Release | ON | 仅在声明支持 Win32 时执行完整构建和布局检查。 |

草案配置的标准命令如下；OFF 配置使用同一命令并将开关设为 `OFF`：

```powershell
conan install . --output-folder=build/conan-release --profile=profiles/windows-msvc-release --lockfile=conan.lock --build=never
cmake --preset Release -DYICAD_ENABLE_PLUGIN_ABI_V3_DRAFT=ON "-DCMAKE_PREFIX_PATH=$env:Qt5_DIR"
cmake --build --preset Release
cmake --install build/Release --config Release

conan install . --output-folder=build/conan-debug --profile=profiles/windows-msvc-debug --lockfile=conan.lock --build=never
cmake --preset Debug -DYICAD_ENABLE_PLUGIN_ABI_V3_DRAFT=ON "-DCMAKE_PREFIX_PATH=$env:Qt5_DIR"
cmake --build --preset Debug
cmake --install build/Debug --config Debug
```

人工验收使用安装目录中的程序和现有插件，覆盖 v1/v2 插件加载、v3 demo 导入、回滚、撤销、
文档关闭和插件卸载，并检查资源、文字、块、标注、填充和图像结果。

### 8.7 阶段 8 完成条件

- 声明支持的架构均完成 Debug/Release、草案 OFF/ON 的适用构建与安装；
- 生产静态断言完整，v1/v2 前缀和现有插件加载没有回归；
- 常规 SDK/demo 代码不读取函数表，不填写 `structSize`、`byteStride` 或借用指针；
- 每个公开函数都有中文 Doxygen 注释，说明线程、所有权、参数、句柄和失败语义；
- 支持、降级和拒绝策略与宿主行为一致；
- 构建日志、人工验收结果、代码审查结论和支持架构形成可审查记录；
- 所有 ABI 布局、输入解码和生命周期问题均已关闭。

建议提交边界：

1. `harden plugin abi v3 boundaries`；
2. `finalize plugin abi v3 release candidate`。

## 6. 阶段 9：正式发布 ABI v3

目标：把已通过阶段 8 的同一候选布局转为正式 ABI。此阶段禁止新增能力、重排字段或改变
失败语义。

### 9.1 冻结名称和版本

在同一提交中完成：

- 定义 `YICAD_PLUGIN_ABI_V3 UINT32_C(3)`；
- 将 `YICAD_PLUGIN_ABI_MAX_VERSION` 和 `YICAD_PLUGIN_ABI_VERSION` 提升为 v3；
- 将 `YICAD_HOST_API_V3_DRAFT_SIZE`、`YICAD_IMPORT_API_V3_DRAFT_SIZE` 改为正式名称；
- 将头文件、SDK、宿主和 demo 中的 `YICAD_PLUGIN_ABI_V3_DRAFT` 改为正式常量；
- 删除 `YICAD_ENABLE_PLUGIN_ABI_V3_DRAFT` 条件编译和 CMake 选项，使 v3 成为默认宿主能力；
- 保留 v1/v2 的版本常量、大小常量和协商分支，不做清理式重构。

草案从未对外发布，不需要为 `_DRAFT` 名称提供兼容别名；避免把临时名称带入正式契约。

### 9.2 固化公开契约和交付物

- 把阶段 8 确认的 Win32/Win64（或最终声明架构）布局断言改为正式 v3 快照；
- 在 `PLUGIN_ABI_EVOLUTION.md` 增加正式 v3 协商矩阵、扩展规则和发布基线；
- 更新 `PLUGIN_SDK.md`，删除草案说明，以高层 SDK 作为首选用法；
- 新增或整理正式 ABI v3 参考文档，逐项记录函数、结构、默认值、所有权和错误码；
- 更新 demo，使其默认声明 v3，同时保留协商到 v2 后的明确兼容路径；
- 安装公开头文件、CMake package、SDK 指南、ABI 契约和可独立构建的 demo；
- 不再把本实施计划作为 SDK 契约安装；历史计划保留在源码仓库即可；
- 按项目发布策略更新 SDK/应用版本和发布说明，明确支持架构及延期能力。

### 9.3 发布验证

从干净构建目录重新执行阶段 8 的构建、安装、静态检查和人工验收，并额外确认：

- 不定义任何草案宏即可构建 v3 宿主和 demo；
- 安装树公开头文件包含正式 v3，源码树私有路径不是外部插件的隐式依赖；
- 仓库外示例只通过安装的 `YiCAD::PluginSdk` 成功构建；
- `dumpbin /exports` 可见且仅依赖约定的三个插件入口名；
- v1、v2、v3 插件均按预期协商并运行；
- 安装包不包含 `_DRAFT` 常量、草案开关或声称 v3 未发布的文案；
- Runtime 和 PluginSDK 组件安装内容、许可证及文档完整。

发布前全局检查：

```powershell
rg -n "V3_DRAFT|YICAD_ENABLE_PLUGIN_ABI_V3_DRAFT|ABI v3 仍是未发布|ABI v3 尚未发布" `
  YiCAD examples CMakeLists.txt doc/PLUGIN_SDK.md doc/PLUGIN_ABI_EVOLUTION.md
```

命令必须无匹配。历史计划可以保留草案记录，但不得纳入扫描范围或安装包。

### 9.4 发布冻结

记录最终提交、编译器、架构、依赖版本、构建命令和验收结果，并以该提交作为 ABI v3
发布基线。自此：

- 既有公开字段、函数签名、调用约定、枚举值和语义永久冻结；
- 兼容扩展只能在明确可扩展结构或函数表尾部追加字段，并提升后续 ABI 版本；
- 任何破坏性变化必须发布新 ABI，不能修改 v3 原位修复；
- 发布后发现的兼容性缺陷若无法保持 v3 契约，只能禁用相关能力或转入新版本设计。

阶段 9 的建议单一提交为 `release plugin abi v3`。在该提交前后分别保留可复现的阶段 8
验证记录，避免发布操作与候选行为修改混在一起。

## 7. 最终完成定义

ABI v3 只有同时满足以下条件才算完成：

1. 阶段 8 的静态检查、构建、安装、审查和人工门禁全部通过；
2. 正式 SDK 默认公开 v3，且 v1/v2 插件继续工作；
3. 常规插件代码不接触 ABI 记账字段，裸 C ABI 仍具备可验证的兼容扩展机制；
4. 成功导入为单个可撤销操作，所有失败和清理路径不留下部分文档状态；
5. 安装 SDK 可在仓库外独立消费，公开文档与实现一致；
6. 最终 ABI 基线已记录，后续修改受演进规则约束。
